# Domain Model

This document describes the core data structures of the scheduling system.

The domain layer contains **pure data objects**.
No business logic or side effects are implemented here.

The goal of the domain model is to clearly represent:

- What the user wants to do (Tasks)
- How time is allocated (Assignments)
- How the user modifies the timeline (Exceptions)
- The overall planning result (Schedule)

---

## Design Principle

The system separates:

- **Requirement** (Task)
- **Concrete allocation** (Assignment)
- **User intervention** (Exception)
- **Planning result snapshot** (Schedule)

Time is modeled using fixed 30-minute atomic units (`TimeBlock`).
This simplifies scheduling logic and ensures deterministic behavior.

---

## Core Objects

### 1. Task

Represents a user requirement.

A task defines *what must be completed* and within *which time window*,
but it does not contain any scheduling result.

Fields:

- `id`
- `title`
- `window_start`
- `window_end`
- `estimate_hours`
- `priority` (optional)

---

### 2. Assignment

Represents a concrete time allocation for a task.

A single task may correspond to multiple assignments.

Fields:

- `task_id`
- `start`
- `end`

---

### 3. Exception

Represents user intervention on the timeline.

Types:

- `BLOCK` — time is unavailable
- `LOCK` — an assignment is fixed and cannot be moved

Fields:

- `task_id` (optional, depending on type)
- `start`
- `end`
- `type`

---

### 4. Schedule

Represents the full planning result.

The schedule is a snapshot generated by the planning service.

Fields:

- `assignments`
- `generated_time`
- `version`

Responsibilities:

- UI display
- Version tracking
- Exporting
- Saving/loading

---

## Supporting Object

### TimeBlock

Represents the smallest unit of time in the system (30 minutes).

Fields:

- `start`
- `end`
- `status`  
  - `FREE`
  - `BLOCKED`
  - `ASSIGNED`
  - `LOCKED`
- `task_id` (optional)

The scheduling algorithm operates on `TimeBlock`s.

---

## Object Relationships

- A `Task` defines a requirement.
- A `Task` may have multiple `Assignment`s.
- A `Schedule` contains many `Assignment`s.
- `Exception`s modify how time blocks are treated during planning.
- The planner converts tasks into assignments using atomic `TimeBlock`s.
